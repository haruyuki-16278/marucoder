# 仕様書: marucoder (初版)

最終更新日: 2026-06-25

## 1. 背景と目的

本システムは、AtCoderの提出・判定体験を簡易再現する学習向けWebアプリである。
ユーザーは C 言語でソルバーを作成・提出し、バックエンドでコンパイルと実行を行い、テストケース単位の判定結果を受け取る。

目的:
- 競技プログラミングの提出サイクルを手元で反復練習できること
- 問題作成者が複数テストケースによる自動判定を提供できること
- 判定過程の可観測性 (コンパイルエラー、実行時間、標準出力差分) を確保すること

## 2. 開発方針 (SDD)

本プロジェクトは SDD (Specification-Driven Development) で進める。

実施方針:
- 仕様を先に定義し、仕様に対応する受け入れ条件を明示する
- 実装前に API 契約とデータ構造を固定し、段階的に実装する
- 実装変更は仕様書と差分確認し、仕様の更新を必須とする

ドキュメント運用:
- 本書は「プロダクト仕様 (What)」を定義する
- 詳細設計は別ドキュメントで管理する
- タスク化時は本書の受け入れ条件に紐付ける

## 3. スコープ

### 3.1 対象 (MVP)

- 問題一覧表示
- 問題詳細表示 (問題文、制約、入出力仕様、サンプル)
- グループ登録/所属 (授業単位)
- 教卓向け進捗ダッシュボード表示
- C 言語ソース提出
- 提出の非同期ジャッジ実行
- 判定結果表示 (AC/WA/TLE/RE/CE)
- 提出履歴表示

### 3.2 対象外 (MVPでは実施しない)

- 複数言語対応 (C 以外)
- ランキング、コンテスト機能
- 高度な差分可視化 UI
- 本番レベルの分散ジャッジ基盤

## 4. ユーザーとユースケース

### 4.1 ユーザー種別

- 学習者: 問題を解いて提出・結果確認する
- 教員: グループ作成、学生所属管理、学習状況確認を行う
- 問題作成者/管理者: 問題とテストケースを登録する

### 4.2 主要ユースケース

1. 学習者が問題ページを開く
2. C ソースを貼り付けて提出する
3. システムがコンパイルして実行する
4. システムが各テストケースを判定し、最終結果を返す
5. 学習者が提出履歴から過去結果を確認する

## 5. 機能要件

### 5.1 問題管理

- 問題は以下を持つ
	- 問題 ID
	- タイトル
	- 問題文 (Markdown)
	- 入力仕様
	- 出力仕様
	- 制約
	- サンプル入出力
	- 公開状態

### 5.2 テストケース管理

- 各問題に複数テストケースを紐付ける
- テストケースは以下を持つ
	- ケース ID
	- 入力データ
	- 期待出力
	- 公開可否 (表示用/隠しケース)

### 5.3 提出機能

- 提出時に以下を受け付ける
	- 問題 ID
	- 言語 (MVP は c 固定)
	- ソースコード
- 提出を受理したら submission ID を返す
- 判定キューに投入する

### 5.4 判定機能

- コンパイル
	- コマンド: gcc
	- コンパイル失敗時は CE を返す
- 実行
	- 各テストケースを独立実行
	- 制限時間超過は TLE
	- 異常終了は RE
- 出力比較
	- デフォルトは厳密一致
	- 行末空白と末尾改行の扱いは比較ポリシーで定義

### 5.5 結果表示

- 総合結果を表示 (AC/WA/TLE/RE/CE)
- ケースごとの結果を表示
- 実行時間、終了コード、標準エラーの要約を表示

### 5.6 提出履歴

- ユーザーごとに提出一覧を表示
- フィルタ: 問題 ID、結果、提出日時

### 5.7 グループ登録/所属管理

- 授業で使用するグループ (例: 1年A組、1年B組) を作成できる
- グループは以下を持つ
	- グループ ID
	- グループ名
	- 登録コード (参加用)
	- 作成者 (教員)
	- 開始日/終了日 (任意)
- 学習者は登録コードを使ってグループ参加できる
- 学習者は同時に 1 つのアクティブグループに所属する (MVP)
- 教員はグループ内メンバー一覧と提出状況を閲覧できる
- 教員は誤参加ユーザーをグループから除外できる

### 5.8 教卓向け進捗ダッシュボード

- 教員は教卓表示モードで以下を同時表示できる
	- グループごとの進捗テーブル
	- 席順ベースの学生進捗マップ
- グループ進捗テーブルの表示項目
	- グループ名
	- 所属人数
	- 課題ごとの AC 人数 / 提出人数
	- グループ進捗率 (AC 人数 ÷ 所属人数)
	- 最終提出時刻
- 席順マップの表示要件
	- 教室レイアウト (行・列) に沿って学生を配置
	- 各席に学生氏名、最新判定、提出回数、最終更新時刻を表示
	- 判定状態を色で識別 (例: AC=緑, WA=橙, CE/RE/TLE=赤, 未提出=灰)
	- 空席を区別して表示
- 表示更新は自動リフレッシュ (MVP: 10 秒間隔ポーリング)
- 教員は問題を切り替えて進捗を確認できる

### 5.9 席順管理

- グループごとに席順を登録・更新できる
- 席情報は以下を持つ
	- 行番号、列番号
	- 学生ユーザー ID (未割当可)
	- 表示ラベル (任意)
- 学生の転席に対応するため、授業中に再配置可能とする

## 6. 非機能要件

### 6.1 性能

- 提出受付 API の応答: p95 300ms 以内 (キュー投入まで)
- 1 提出あたりの判定完了目標: 10 秒以内 (ケース数・制約による)

### 6.2 可用性

- 開発/検証用途のため、常時稼働を必須としない
- 判定中断時に再実行可能であること

### 6.3 セキュリティ

- 実行プロセスを隔離する (将来: コンテナ/サンドボックス)
- CPU 時間、メモリ、ファイルアクセスを制限する
- 任意の外部ネットワークアクセスを禁止する
- ソースコード・入出力ログに機密情報を含めない運用を徹底する
- 登録コードは推測されにくいランダム文字列で発行する
- グループ内の提出データは所属メンバー/教員のみ閲覧可能とする

### 6.4 監査性

- 提出、コンパイル、実行、判定結果を時系列で記録する
- 障害調査用に最小限ログを保持する

## 7. 判定ステータス定義

- AC: 全ケース正解
- WA: 実行は完了したが期待出力と不一致
- CE: コンパイル失敗
- RE: 実行時エラー (シグナル終了、異常終了)
- TLE: 制限時間超過
- IE: システム内部エラー (MVPでは暫定)

最終判定の優先順位 (複数ケース時):
1. CE
2. TLE
3. RE
4. WA
5. AC

## 8. システム構成 (論理)

構成要素:
- Frontend: 問題表示、コード入力、提出、結果表示
- API: 提出受付、結果取得、問題取得
- Judge Worker: コンパイル・実行・比較
- Storage: 問題、テストケース、提出、結果

判定フロー:
1. ユーザーが提出 API を呼ぶ
2. API が提出レコードを保存し、キューに投入
3. Worker が提出を取得してコンパイル
4. 成功時は全ケース実行、比較、結果保存
5. API からクライアントへ結果を返却

## 10. API 仕様 (MVP草案)

### 10.1 問題一覧取得

- Method: GET
- Path: /api/problems
- Response: ProblemSummary[]

### 10.2 問題詳細取得

- Method: GET
- Path: /api/problems/:problemId
- Response: ProblemDetail

### 10.3 提出作成

- Method: POST
- Path: /api/submissions
- Request:
	- problemId: string
	- language: "c"
	- sourceCode: string
- Response:
	- submissionId: string
	- status: "QUEUED"

### 10.4 提出結果取得

- Method: GET
- Path: /api/submissions/:submissionId
- Response:
	- submission
	- caseResults

### 10.5 提出一覧取得

- Method: GET
- Path: /api/submissions?problemId=&verdict=
- Response: SubmissionSummary[]

### 10.6 グループ作成

- Method: POST
- Path: /api/groups
- Request:
	- name: string
	- startsAt?: string
	- endsAt?: string
- Response:
	- groupId: string
	- joinCode: string

### 10.7 グループ参加

- Method: POST
- Path: /api/groups/join
- Request:
	- joinCode: string
- Response:
	- groupId: string
	- role: "student"

### 10.8 グループメンバー一覧取得

- Method: GET
- Path: /api/groups/:groupId/members
- Response: GroupMemberSummary[]

### 10.9 グループ別提出一覧取得 (教員用)

- Method: GET
- Path: /api/groups/:groupId/submissions?problemId=&userId=&verdict=
- Response: SubmissionSummary[]

### 10.10 グループ進捗テーブル取得 (教卓用)

- Method: GET
- Path: /api/dashboard/groups?problemId=
- Response: ProgressSnapshot[]

### 10.11 席順マップ取得 (教卓用)

- Method: GET
- Path: /api/dashboard/groups/:groupId/seats?problemId=
- Response: SeatProgress[]

### 10.12 席順登録/更新 (教員用)

- Method: PUT
- Path: /api/groups/:groupId/seats
- Request:
	- seats: Array<{ row: number; col: number; userId?: string; label?: string }>
- Response:
	- updatedCount: number

### 10.13 席順 CSV インポート (教員用)

- Method: POST
- Path: /api/groups/:groupId/seats/import
- Content-Type: text/csv
- Request:
	- CSV ファイル本体 (17章フォーマット)
- Response:
	- acceptedCount: number
	- updatedCount: number
	- errorCount: number
	- errors: Array<{ line: number; reason: string }>

## 11. エラーハンドリング

- 400: 入力不正 (必須項目欠落、言語不正)
- 404: 問題または提出が見つからない
- 409: 判定中など状態競合
- 500: サーバ内部エラー

エラーレスポンス形式:
- code: string
- message: string
- details?: string

## 12. 受け入れ条件 (MVP)

- 正常系
	- C ソース提出で AC が返る
	- 複数ケースで 1 ケース不一致時に WA が返る
- 異常系
	- コンパイルエラーで CE が返る
	- 無限ループ系入力で TLE が返る
	- 0 除算などで RE が返る
- 表示
	- 提出詳細に総合結果とケース別結果が表示される
- グループ
	- 教員がグループ作成すると登録コードが発行される
	- 学習者が登録コードで参加できる
	- 教員は所属メンバーと提出状況を閲覧できる
	- 教卓画面でグループ別進捗テーブルを表示できる
	- 教卓画面で席順に応じた学生進捗を表示できる

## 17. 席順 CSV フォーマット定義

### 17.1 目的

- 教員が名簿と席順を事前に作成し、一括登録できるようにする

### 17.2 文字コードと区切り

- 文字コード: UTF-8
- 改行: LF または CRLF
- 区切り文字: カンマ (,)
- 1 行目はヘッダ必須

### 17.3 カラム定義

- groupId (必須): グループ ID
- row (必須): 1 以上の整数
- col (必須): 1 以上の整数
- userId (任意): 学生ユーザー ID
- studentName (任意): 表示名
- label (任意): 座席ラベル

### 17.4 CSV 例

```csv
groupId,row,col,userId,studentName,label
group_a,1,1,s001,山田 太郎,01
group_a,1,2,s002,佐藤 花子,02
group_a,1,3,,,03
group_a,2,1,s003,鈴木 一郎,04
```

### 17.5 バリデーション

- 同一 groupId 内で (row, col) の重複を禁止
- row, col は 1 以上 100 以下 (MVP)
- userId が指定される場合、当該 groupId のメンバーであること
- 空席は userId と studentName を空欄にする

### 17.6 インポート結果

- 受理件数、更新件数、エラー件数を返す
- エラー行は行番号と理由を返す

## 19. 実装タスク分解 (MVP)

### 19.1 バックエンド

1. Group / GroupMember / ClassroomSeat の永続化実装
2. 既存 Submission に groupId を追加し、登録時に関連付け
3. GET /api/dashboard/groups を実装 (問題別進捗集計)
4. GET /api/dashboard/groups/:groupId/seats を実装 (席順進捗)
5. PUT /api/groups/:groupId/seats を実装 (手動更新)
6. POST /api/groups/:groupId/seats/import を実装 (CSV取込)
7. CSV バリデータ実装 (重複座標、範囲、メンバー整合)

### 19.2 フロントエンド

1. 教員向け 教卓進捗画面を追加
2. グループ進捗テーブルコンポーネントを作成
3. 席順マップコンポーネントを作成
4. 10秒ポーリングと問題切替連動を実装
5. 席セルクリックで学生提出履歴へ遷移実装
6. 席順編集 UI と CSV アップロード UI を実装

### 19.3 テスト

1. 進捗集計 API のユニットテスト
2. 席順更新 API のバリデーションテスト
3. CSV インポート API の正常系/異常系テスト
4. 教卓画面の結合テスト (課題切替、自動更新、席クリック)

### 19.4 受け入れ確認

1. 教員がグループ作成から席順登録まで完了できる
2. 学生提出後、10秒以内に教卓画面へ反映される
3. 不正 CSV を投入した際、行番号付きエラーが返る
4. 教卓画面が氏名表示で進捗確認できる
